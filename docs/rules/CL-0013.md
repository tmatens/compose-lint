# CL-0013: Sensitive Host Path Mounted

**Severity:** HIGH (CRITICAL when the entire host root filesystem is mounted)

**References:**
- [OWASP Docker Security Rule #8](https://cheatsheetseries.owasp.org/cheatsheets/Docker_Security_Cheat_Sheet.html#rule-8---set-filesystem-and-volumes-to-read-only)
- CIS Docker Benchmark 5.5 ‚Äî Do not mount sensitive host system directories

## What it detects

Services that bind-mount sensitive host directories into a container:
- `/` ‚Äî the entire host root filesystem (CRITICAL ‚Äî one-line container escape)
- `/etc` ‚Äî host system configuration
- `/proc` ‚Äî kernel and process information
- `/sys` ‚Äî kernel device and driver interfaces
- `/boot` ‚Äî bootloader and kernel images
- `/root` ‚Äî root user's home directory (SSH keys, shell history)
- `/var/lib/docker` ‚Äî Docker daemon state, including images and other containers
- `/var/run` ‚Äî runtime sockets (parent of `/var/run/docker.sock`)
- `/home` ‚Äî user home directories with credentials and source code

Subpaths (e.g., `/etc/passwd`, `/root/.ssh`) are also flagged. Both short syntax (`/etc:/host-etc`) and long syntax (`source: /etc, target: /host-etc`) are checked. The long syntax is recognized whether `type: bind` is set explicitly or omitted ‚Äî Compose treats absolute-path sources as bind mounts either way. Named volumes are not flagged.

## Why it matters

Mounting these paths gives the container read (and potentially write) access to critical host infrastructure:
- `/` is a complete container escape ‚Äî the workload has full access to the host filesystem
- `/etc` contains `shadow`, `sudoers`, `ssh` configs ‚Äî a writable mount enables full host takeover
- `/proc` exposes kernel parameters, process environments (secrets), and sysrq triggers
- `/sys` provides direct interfaces to hardware, cgroups, and kernel modules
- `/boot` contains the kernel and initramfs ‚Äî modification enables persistent rootkits
- `/root` and `/home` typically contain SSH keys, cloud credentials, and shell history
- `/var/lib/docker` exposes other containers' filesystems and image layers
- `/var/run` is the parent of the Docker socket and other privileged sockets

## Fix

Remove the bind mount. If the container needs specific host files, copy them into the image at build time or use a named volume containing only the required data:

```yaml
# Before
services:
  app:
    image: myapp:1.0
    volumes:
      - /etc:/host-etc

# After ‚Äî use a named volume or COPY in Dockerfile
services:
  app:
    image: myapp:1.0
    volumes:
      - app-config:/app/config
```
